## 4. 精确修复清单

## 4.1 P0（必须本轮完成）

### P0-01 去钱包化铸造与重试链路改造
**问题定义**
- 前端在单铸、重试中仍依赖 `connect()` 或 `localStorage.wallet_address`。
- 批量铸造确认弹窗强制输入钱包地址。
- 后端请求体 `minter_address` 为必填，导致“无钱包设置”场景主流程受阻。

**修改动作**
- 前端：将“接收地址来源”改为系统策略（企业配置地址或后端默认地址），不再要求钱包连接。
- 前端：移除“请先连接钱包”类阻断提示，改为“请先配置企业接收地址/使用系统默认地址”。
- 后端：`mint/retry/batch-mint/estimate` 支持 `minter_address` 省略；按统一优先级回填：`request.minter_address` > 企业 `wallet_address` > 系统 `DEPLOYER_ADDRESS`。
- 后端：若三者均为空，返回统一业务错误码 `MINTER_ADDRESS_NOT_CONFIGURED`。

**验收标准**
- 在不连接钱包、不签名条件下，单铸/批铸/重试可成功触发并返回真实 `token_id`、`tx_hash`、`status`。
- 前端不再出现“连接钱包后才能铸造”的阻断交互。
- 后端错误码可被前端稳定识别并提示。

### P0-02 API 文档与路径定义最终对齐
**问题定义**
- 后端存在主路径 + alias（deprecated）并存，文档章节仍有过期描述。

**修改动作**
- 固化主路径为：`/api/v1/nft/mint`、`/api/v1/nft/{asset_id}/mint/status`、`/api/v1/nft/{asset_id}/mint/retry`、`/api/v1/nft/mint/history`、`/api/v1/nft/{token_id}/history`。
- 明确 alias 下线计划（版本、时间、影响范围）。
- 修订需求文档中的状态门槛描述，统一为 `APPROVED` 才可铸造。

**验收标准**
- 前端调用路径、后端主路由、文档三方一致。
- alias 保留为兼容路径且标注 `deprecated` 与下线计划。

## 4.2 P1（应在本轮完成）

### P1-01 批量铸造前端 50 条限制与可视化反馈
**修改动作**
- 前端选择资产时增加 `<=50` 硬性校验。
- 超限时在确认前阻断并提示可操作文案。

**验收标准**
- 选择 51 条时无法提交且提示明确。
- 选择 50 条及以下可正常提交。

### P1-02 批量任务状态细化
**修改动作**
- 结果面板增加“进行中/成功/失败”统计与逐条状态。
- 失败项保留重试入口（跳转单条重试或直接触发重试）。

**验收标准**
- 用户可定位失败资产及失败原因。
- 失败资产可从同一页面继续重试。

### P1-03 错误码映射与提示统一
**修改动作**
- 统一 NFT/IPFS 关键错误码到前端提示映射。
- 补齐 `MINTER_ADDRESS_NOT_CONFIGURED`、`INSUFFICIENT_ATTACHMENTS`、`CONTRACT_CALL_FAILED` 等提示文案。

**验收标准**
- 主流程关键错误均有稳定、可执行提示。

## 4.3 P2（增强与治理）

### P2-01 NFT 路由重复配置治理
**修改动作**
- 合并 `router.tsx` 与 `router/routes/nft.tsx` 的重复配置，保留单一事实源。
- 清理遗留入口，降低后续维护风险。

**验收标准**
- `/nft/dashboard`、`/nft/assets`、`/nft/minting`、`/nft/history`、`/nft/contracts` 路由稳定可达。
- 菜单、面包屑、实际路由配置一致。

### P2-02 合约事件字段对齐增强
**修改动作**
- 统一事件字段（含时间戳）在后端响应和前端展示层的字段命名与映射。

**验收标准**
- 历史明细字段可追溯，显示一致，无字段缺失。

## 5. 前后端改动评估结论

### 5.1 前端是否还需要修改
- 需要修改（高优先）：去钱包化交互、批量50条限制、批量状态细化、错误码提示映射。
- 可延后修改（低优先）：NFT 路由重复配置治理。

### 5.2 后端是否还需要修改
- 需要修改（高优先）：`minter_address` 可选化与默认地址回填策略、无可用地址时统一错误码。
- 本轮可不改：签名验签强化链路（按“去钱包化”前提暂缓）。
