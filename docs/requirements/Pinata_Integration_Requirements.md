# Pinata 集成与存储链路统一需求文档

## 1. 文档信息

| 项目 | 内容 |
|------|------|
| 文档版本 | v1.0 |
| 创建日期 | 2026-03-16 |
| 适用范围 | Web3 IP-NFT 资产管理系统（前端/后端） |
| 目标 | 统一附件与元数据上链前存储链路，提升安全性与一致性 |

---

## 2. 背景与问题

### 2.1 当前现状
- 前端存在 `pinata` SDK 封装，依赖 `VITE_PINATA_JWT_TOKEN` 进行直传能力预留。
- 后端存在 `PinataService`，并提供 `/api/v1/ipfs/*` 接口。
- 资产创建页面当前只提交资产基础字段，附件选择后仅写入本地表单元数据，未实际上传。
- NFT 铸造元数据当前由后端 `get_ipfs_client()` 上传到本地 Kubo（`IPFS_API_URL`），不走 Pinata。
- 系统存在“附件可能走 Pinata、metadata 走本地 Kubo”的双通道并存状态。

### 2.2 核心问题
- 存储链路不统一，排障和审计成本高。
- 前端暴露 `VITE_PINATA_JWT_TOKEN` 存在凭证风险。
- 文件校验规则不一致（前端与后端限制存在差异）。
- 业务流程与接口能力不匹配（有 `with-attachments` 接口但前端未接入主流程）。

---

## 3. 目标与范围

### 3.1 目标
- 将 Pinata 相关能力统一收敛到后端，前端不直接持有高权限凭证。
- 打通资产创建与附件上传闭环，确保附件生成真实 CID 并入库。
- 明确 metadata 上传策略，避免附件与 metadata 存储分裂。
- 建立统一校验和错误语义，保障前后端行为一致。

### 3.2 范围
- 前端：资产创建流程、附件上传交互、错误提示与状态展示。
- 后端：Pinata 服务层、资产与 IPFS 相关 API、NFT metadata 上传策略。
- 配置：环境变量与多环境参数规范。

### 3.3 非范围
- 不涉及智能合约 ABI 变更。
- 不涉及链上业务规则重定义。
- 不引入新的对象存储供应商。

---

## 4. 架构原则

- 安全优先：敏感凭证仅保留在服务端。
- 单一主链路：同一业务域尽量使用同一存储入口。
- 最小改动：优先复用现有接口与服务，避免大规模重构。
- 可观测性：上传链路必须具备可追踪日志与错误码。

---

## 5. 功能需求

### FR-001 资产创建接入一体化上传
**优先级**: P0  
**描述**: 前端资产创建流程接入 `POST /api/v1/assets/with-attachments`，实现资产与附件一次提交。  
**验收标准**:
- [ ] 资产创建可携带多文件上传（最多 10 个）。
- [ ] 上传成功后附件表写入真实 `ipfs_cid`。
- [ ] 响应中返回附件数量、总大小与网关基址。

### FR-002 前端取消 Pinata 凭证依赖
**优先级**: P0  
**描述**: 前端主业务流程不再依赖 `VITE_PINATA_JWT_TOKEN`。  
**验收标准**:
- [ ] 资产创建主流程不使用前端直传 Pinata。
- [ ] 前端环境变量中不再要求生产环境配置 Pinata JWT。
- [ ] 上传失败提示改为后端错误语义透传。

### FR-003 统一附件校验规则
**优先级**: P1  
**描述**: 前后端统一文件类型、大小、数量限制。  
**验收标准**:
- [ ] 单文件大小统一为 50MB。
- [ ] 支持文件类型与后端白名单完全一致。
- [ ] 超限与格式错误在前端预校验并在后端兜底校验。

### FR-004 统一 metadata 存储策略
**优先级**: P1  
**描述**: NFT metadata 上传策略明确为单一通道（推荐后端 Pinata）。  
**验收标准**:
- [ ] metadata 上传入口与附件存储策略一致。
- [ ] `metadata_uri` 生成与可访问性符合现有铸造逻辑。
- [ ] 失败时保留现有铸造状态机的错误回写能力。

### FR-005 兼容保留分步接口
**优先级**: P2  
**描述**: 保留 `/api/v1/ipfs/*` 与 `/assets/{id}/attachments` 以兼容历史流程。  
**验收标准**:
- [ ] 旧接口可用且不影响新主流程。
- [ ] 文档中标注推荐路径与兼容路径。

---

## 6. 非功能需求

### NFR-001 安全性
- Pinata JWT 仅允许服务端读取。
- 日志禁止输出完整凭证。
- 删除、上传接口保持认证与鉴权策略。

### NFR-002 性能
- 单文件上传接口在 50MB 下稳定处理。
- 一体化创建请求在正常网络下有可接受响应时间。

### NFR-003 可靠性
- 上传失败时需保证资产/附件数据一致性（已有回滚逻辑保持可用）。
- Pinata 请求失败需具备重试与明确错误信息。

### NFR-004 可维护性
- 统一错误码与错误文案，便于前端展示与定位。
- 关键链路具备结构化日志字段（asset_id、cid、file_name）。

---

## 7. 接口与流程要求

### 7.1 推荐主流程
1. 前端收集资产表单 + 文件列表。  
2. 调用 `POST /api/v1/assets/with-attachments`。  
3. 后端校验企业成员与文件规则。  
4. 后端通过 `PinataService` 上传文件并落库附件 CID。  
5. 返回资产与附件摘要。  

### 7.2 兼容流程
- 分步流程 A：`POST /assets` → `POST /ipfs/upload` → `POST /assets/{asset_id}/attachments`。  
- 分步流程 B：`POST /assets` → `POST /assets/{asset_id}/attachments`（使用已有 CID）。  

### 7.3 metadata 流程约束
- 铸造前 metadata 必须完成上传并生成 `ipfs://` URI。
- metadata 上传失败需中断铸造并写入失败原因。

---

## 8. 数据与配置需求

### 8.1 环境变量
**后端必需**
- `PINATA_JWT_TOKEN`（推荐）
- `PINATA_API_KEY` / `PINATA_API_SECRET`（兼容）

**前端**
- 不要求主流程配置 Pinata JWT。
- 保留 `VITE_PINATA_GATEWAY_URL` 仅用于展示网关 URL 拼接。

### 8.2 数据字段
- 附件表必须持久化 `ipfs_cid`、`file_name`、`file_type`、`file_size`。
- 资产表必须持久化 `metadata_cid`、`metadata_uri`（用于铸造与查询）。

## 9. 里程碑与实施计划

### M1（P0）
- 前端接入 `with-attachments` 主流程。
- 前端去除主流程 Pinata JWT 依赖。
- 校验规则统一到 50MB 与统一类型白名单。

### M2（P1）
- metadata 上传通道统一为 Pinata 或明确保留 Kubo 并形成固定策略。
- 完成链路错误码与日志字段标准化。

### M3（P2）
- 输出接口使用规范与兼容路径下线计划。

---

## 10. 验收标准

- [ ] 从资产创建到附件落库可在一次请求内完成。
- [ ] 附件记录均带有效 `ipfs_cid`。
- [ ] 前端生产环境不再要求 Pinata JWT 才能完成主流程。
- [ ] metadata 上传策略与附件策略一致或有明确架构决策记录。
- [ ] 失败场景（超限、类型不支持、Pinata 超时）有稳定提示与后端日志。

---

## 11. 风险与缓解

- 风险：Pinata 外部网络抖动导致上传失败。  
  缓解：后端重试、超时配置、错误码标准化。

- 风险：历史流程与新流程并存引起使用混乱。  
  缓解：前端仅暴露主流程入口，兼容接口仅保留给内部调用。

- 风险：metadata 迁移改动影响铸造稳定性。  
  缓解：灰度开关控制，先单资产后批量启用。

---

## 12. 兼容接口使用规范与下线计划（M3）

### 12.1 使用规范
- 前端默认仅使用 `POST /api/v1/assets/with-attachments` 作为资产创建主入口。
- `/api/v1/ipfs/upload` 与 `/api/v1/assets/{asset_id}/attachments` 仅用于历史流程兼容与内部运维场景。
- 新增业务功能不得再接入前端直传 Pinata，不得引入 `VITE_PINATA_JWT_TOKEN` 作为主流程前置依赖。

### 12.2 下线阶段计划
- 阶段一（观测期，2 周）：保留兼容接口，新增日志字段 `route_type=legacy|primary`，统计调用来源与调用量。
- 阶段二（收敛期，2 周）：前端隐藏兼容入口，仅允许内部白名单调用兼容接口。
- 阶段三（冻结期，1 周）：兼容接口返回弃用提示头 `X-API-Deprecated: true`，并在网关告警低频调用。
- 阶段四（下线期）：当兼容调用量连续 7 天为 0 时，执行下线评审并移除兼容入口。

### 12.3 回滚策略
- 若出现 P1 及以上故障，立即恢复兼容接口外部可用性，回退至阶段一观测期策略。
- 回滚窗口内保留主流程优先，不恢复前端 Pinata JWT 依赖。
