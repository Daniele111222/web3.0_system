## 1. 系统总体架构

### 1.1 系统背景与目标

随着区块链技术与Web3生态的快速发展，传统知识产权（IP）管理体系面临以下痛点：权属证明依赖中心化机构、溯源链路不透明、商业化授权流程复杂低效、跨境确权成本高昂。本系统旨在构建一套企业级知识产权去中心化应用（DApp），以ERC-721标准NFT作为IP资产的链上凭证，结合IPFS分布式存储与智能合约自动化执行，实现从IP登记、确权、溯源到商业化的全生命周期管理。

**系统核心目标：**

- 以NFT作为知识产权的数字孪生凭证，实现链上不可篡改的权属证明
- 通过智能合约自动化审批、授权与收益分配，降低人工干预成本
- 利用IPFS实现链下元数据与附件的去中心化持久存储
- 构建完整的链上溯源体系，支持任意时刻的权属验证
- 为企业提供多链部署能力，适配Ethereum、Polygon、BSC等主流EVM链

## 2. 用户认证与身份管理模块

### 2.1 模块概述

该模块负责用户注册、登录、邮箱验证、密码管理、企业绑定及区块链钱包绑定等功能，是整个系统的安全基础。

### 2.2 前端需求

#### 2.2.1 页面与路由

| 路由 | 页面 | 说明 |
|------|------|------|
| `/auth` | 登录/注册页 | 支持选项卡切换登录和注册表单 |
| `/auth/forgot-password` | 忘记密码页 | 输入邮箱发送重置链接 |
| `/auth/reset-password` | 重置密码页 | 通过token链接设置新密码 |

#### 2.2.2 功能需求

**登录表单：**
- 邮箱+密码登录
- 表单前端验证（邮箱格式、密码非空）
- 登录成功后将 `accessToken` 和 `refreshToken` 持久化到 localStorage
- 登录后重定向至 `/dashboard`

**注册表单：**
- 填写字段：邮箱、用户名、密码、确认密码、所属企业名称
- 密码强度检验（最少8位，包含字母和数字）
- 注册成功后提示验证邮箱

**钱包绑定：**
- 在用户设置页提供"绑定MetaMask钱包"按钮
- 调用 `window.ethereum` 请求连接授权
- 用户对服务端下发的随机消息进行签名
- 将 `walletAddress` + `signature` 提交后端验证
- 绑定成功后在导航栏显示已连接钱包地址（截断格式：`0x1234...abcd`）

**路由守卫：**
- 所有需要登录的页面（非 `/auth`, `/blockchain-explorer`）使用 `ProtectedRoute` 组件包裹
- 未登录访问受保护路由时自动重定向至 `/auth`
- Token过期（401响应）时自动清除本地存储并重定向

**状态管理（Zustand）：**
```typescript
interface AuthState {
  user: User | null
  accessToken: string | null
  refreshToken: string | null
  isAuthenticated: boolean
  // Actions
  login: (credentials) => Promise<void>
  logout: () => void
  refreshAccessToken: () => Promise<void>
}
```

#### 2.3.3 业务逻辑

- JWT访问令牌有效期30分钟，刷新令牌有效期7天
- 密码使用bcrypt哈希存储，work factor ≥ 12
- 钱包绑定时后端验证 `ecrecover(message, signature) == walletAddress`
- 邮箱验证token有效期24小时，密码重置token有效期1小时
- 同一邮件地址60秒内最多发送1次验证邮件（限流）
- 接口限流：登录接口每IP每分钟最多10次请求

### 2.4 区块链需求

- 无直接链上操作；钱包地址仅在链下验证后存储于数据库
- 钱包签名验证逻辑：后端使用 `web3.eth.account.recover_message()` 恢复签名地址
- 后续所有链上操作均需用户钱包地址与数据库记录匹配

#### 3.3.3 业务逻辑

**IPFS上传流程：**
1. 接收文件，验证类型和大小（最大50MB）
2. 调用Kubo节点API上传文件，获取CID
3. 调用Pinata API固定(pin)该CID，确保持久性
4. 返回CID和Pinata gateway URL

**NFT铸造流程（后端编排）：**
1. 验证资产状态为 `approved`
2. 构建NFT元数据JSON（含所有结构化字段和附件CIDs）
3. 上传元数据JSON至IPFS，获取 `metadata_cid`
4. 使用后端私钥调用 `IPNFT.mint(enterpriseWalletAddress, "ipfs://<metadata_cid>")`
5. 等待交易确认（最少1个区块）
6. 更新数据库：`token_id`, `mint_tx_hash`, `status='minted'`
7. 若有版税设置，调用 `setTokenRoyalty(tokenId, receiver, feeNumerator)`

**文件哈希验证：**
- 服务端对上传的代码文件计算SHA-256哈希并记录
- 与客户端上报的哈希值进行比对，确保一致性

## 9. ZKML模块（选做）

> **注意：** 本模块为选做高级扩展，需要专业ZKML框架知识。

### 9.1 模块概述

设计一个"证明者"在链下运行深度学习模型推理，同时使用zk-SNARKs技术为整个计算过程生成密码学证明，将推理结果和ZK Proof提交到智能合约进行链上验证。

### 9.2 系统设计

#### 9.2.1 整体流程

```
输入数据（IP相关特征）
    ↓
[链下]证明者（Prover）
    ├── 运行ONNX/PyTorch推理模型
    ├── 生成电路约束系统（使用ezkl/RISC Zero）
    └── 生成ZK Proof（zk-SNARKs）
    ↓
ZK Proof + 推理结果 + 公开输入
    ↓
[链上]ZKVerifier.sol
    ├── 验证Proof（pairing check）
    └── 若验证通过 → 触发相应业务逻辑
```

#### 9.2.2 应用场景（IP领域）

**场景1：IP相似度检测（防重复注册）**
- 模型：将IP特征（文本描述、图像特征）映射到嵌入向量，计算余弦相似度
- ZK证明：证明"两个IP描述的相似度 < 阈值"而无需暴露IP原始内容
- 链上结果：若证明通过，允许铸造NFT（IP具有足够独特性）

**场景2：IP价值评估（辅助定价）**
- 模型：基于IP类型、权利范围、行业数据的估值模型
- ZK证明：证明估值模型的推理结果正确，无需链上运行模型
- 链上结果：将可信估值记录于Token元数据或许可定价参考

### 9.5 区块链需求

**ZKVerifier.sol 合约：**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

// 由ezkl自动生成的验证器合约
contract ZKVerifier {
    // ezkl生成的验证密钥（硬编码于合约）
    // Groth16或PLONK验证逻辑

    function verifyProof(
        uint256[2] memory a,        // 证明的π_A部分
        uint256[2][2] memory b,     // 证明的π_B部分
        uint256[2] memory c,        // 证明的π_C部分
        uint256[] memory publicInputs  // 公开输入
    ) public view returns (bool)
}

contract IPZKMLOracle is ZKVerifier {
    mapping(uint256 => bool) public verifiedTokens;
    mapping(uint256 => uint256) public inferenceResults;

    // 提交并验证IP相似度证明
    function submitSimilarityProof(
        uint256 tokenId,
        uint256[2] memory a,
        uint256[2][2] memory b,
        uint256[2] memory c,
        uint256[] memory publicInputs  // 含相似度分数
    ) external {
        require(verifyProof(a, b, c, publicInputs), "Invalid ZK Proof");
        uint256 similarityScore = publicInputs[0];
        require(similarityScore < SIMILARITY_THRESHOLD, "IP too similar");
        verifiedTokens[tokenId] = true;
        emit ProofVerified(tokenId, similarityScore);
    }

    event ProofVerified(uint256 indexed tokenId, uint256 similarityScore);
}
```

**推荐ZKML框架：**
- **ezkl**：专为神经网络推理的ZK证明工具，支持ONNX模型，生成Solidity验证器
- **RISC Zero**：通用ZK虚拟机，支持复杂计算证明
- **Circom + snarkjs**：手写电路，灵活但复杂度高
